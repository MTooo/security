## 一、总体架构思路

### 1.1 设计理念：对等系统

两个系统之间数据安全交互时，**双方是对等的**——系统 A 可以调用系统 B，系统 B 也可以调用系统 A。SDK 合并成一个，每个集成方同时具备**主动调用**和**被动响应**两种能力：

```
             主动调用（作为客户端）                被动响应（作为服务端）
             ─────────────────────                ─────────────────────
系统 A       sm2Client.post("/api/pay")           @Sm2Api → 接收 B 的查询请求
系统 B       sm2Client.get("/api/query")          @Sm2Api → 接收 A 的支付请求
```

**一个 SDK，两种角色**：不在代码层面区分"客户端 SDK"和"服务端 SDK"，而是通过配置决定当前系统开启哪些能力。

### 1.2 分层设计

```text
┌─────────────────────────────────────────────────────────────┐
│                      业务应用层                             │
│  主动调用：sm2Client.get() / post() / put() / delete()      │
│  被动响应：@Sm2Api 注解标记 Controller                      │
├─────────────────────────────────────────────────────────────┤
│                      SDK门面层                              │
│  Sm2HttpClient（主动调用）    Sm2ServerFilter（被动响应）    │
├─────────────────────────────────────────────────────────────┤
│                  会话管理模块（核心）                        │
│  - 会话创建（作为客户端时）/ 会话接收（作为服务端时）        │
│  - 密钥缓存（本地Caffeine + Redis双级缓存）                  │
│  - 生命周期管理（时间/次数双维度）                           │
├─────────────────────────────────────────────────────────────┤
│                  SM2密钥协商模块                             │
│  - 临时密钥对生成（每次会话）                                │
│  - 共享密钥计算（椭圆曲线点乘）                              │
│  - KDF密钥派生（SM3哈希迭代）                                │
├─────────────────────────────────────────────────────────────┤
│                  SM4加解密模块                               │
│  - 业务数据加密（GCM模式）                                   │
│  - 认证标签生成/校验                                        │
│  - 会话密钥自动绑定                                          │
├─────────────────────────────────────────────────────────────┤
│                  网络通信层                                  │
│  - 主动调用：Hutool HttpUtil 发送加密请求                    │
│  - 被动响应：Servlet Filter 自动加解密                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 模块依赖关系

- **业务层**只依赖 `Sm2HttpClient`（主动调用）+ `@Sm2Api`（被动响应）
- **会话管理层**依赖**密钥协商模块**和**加解密模块**
- **加解密模块**通过 Hutool 实现 SM2/SM3/SM4
- **网络通信层**：主动调用使用 Hutool `HttpUtil`，被动响应使用 Servlet Filter

## 二、HTTP 协议规范（独立实现参考）

> **如果对方不信任你的 SDK，可以照着本章自己写实现。SDK 只是这份协议的便捷封装，协议本身才是契约。**

### 2.1 整体约定

| 项目 | 约定 |
|------|------|
| 传输协议 | HTTPS（必须） |
| 算法 | SM2 密钥协商（GB/T 32918.3） + SM4-GCM 加解密 |
| 编码 | 所有二进制字段使用 **Base64** 编码传输 |
| 字符编码 | UTF-8 |
| 时间戳 | Unix 毫秒时间戳 |

### 2.2 握手协议（建立会话）

握手是 **3 次 HTTP 请求**，产出一个 `sessionId` 和对应的 SM4 会话密钥。

#### 第一步：客户端 → 服务端

```
POST /sm2/handshake/init
Content-Type: application/json

{
    "protocolVersion": "1.0",
    "clientId": "SYSTEM_A",
    "ephemeralPublicKey": "BASE64(RA)",        // 客户端临时公钥，64字节未压缩
    "timestamp": 1719676800000,
    "signature": "BASE64(SIGN(dA, RA || clientId || ZA || timestamp))"
}
```

#### 第二步：服务端 → 客户端

```
HTTP 200
Content-Type: application/json

{
    "sessionId": "uuid-string",
    "ephemeralPublicKey": "BASE64(RB)",        // 服务端临时公钥
    "confirmation": "BASE64(SB)"               // 服务端确认参数
}
```

#### 第三步：客户端 → 服务端

```
POST /sm2/handshake/confirm
Content-Type: application/json

{
    "sessionId": "uuid-string",
    "confirmation": "BASE64(SA)"               // 客户端确认参数
}
```

```
HTTP 200  → 握手成功，后续可用此 sessionId 发业务请求
HTTP 4xx  → 握手失败，错误体见 2.5
```

### 2.3 业务请求格式（加密调用）

#### 通用格式

所有业务请求（GET / POST / PUT / DELETE）共用同一套加密信封，**HTTP 方法、路径和明文的接口保持一致**，差异仅在于 Body 中加密的明文内容不同。

```
{HTTP方法} /api/xxx（业务路径，和明文的接口路径一样）
Content-Type: text/plain

X-Session-Id: {sessionId}
X-Timestamp: {unixMillis}
X-Nonce: {16字节随机数 Base64}

Body: BASE64( IV[12字节] || CIPHERTEXT || TAG[16字节] )
```

| Header | 必填 | 说明 |
|--------|------|------|
| `X-Session-Id` | 是 | 握手获得的 sessionId |
| `X-Timestamp` | 是 | 请求发起时的毫秒时间戳 |
| `X-Nonce` | 是 | 16 字节随机数，Base64 编码，每次请求必须不同 |

> **Body 结构**：前 12 字节为随机生成的 GCM IV，末尾 16 字节为认证 TAG，中间部分为 SM4-GCM 加密后的密文。接收方用 `X-Session-Id` 查找会话密钥，取前 12 字节作 IV、末 16 字节作 TAG、中间部分作密文进行解密和完整性校验。

---

#### 2.3.1 GET 请求

GET 请求的**查询参数（Query String）序列化为 JSON 后加密**放入 Body。HTTP 方法保持 GET，路径保持不变。

**明文示例**（加密前的 Body）：

```json
{
    "idCard": "110101199001011234",
    "name": "张三",
    "page": 1,
    "size": 20
}
```

**加密后的 HTTP 请求**：

```
GET /api/user/query
Content-Type: text/plain
X-Session-Id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
X-Timestamp: 1719676800000
X-Nonce: RfG3kL9mP2sV7wX1Y5zA==

Body: s8dF2kL9mP3xR7wX1yA5zB6cD0eF4gH8...（Base64 密文，省略）
```

**不带参数的 GET 请求**：Body 为空字符串 `""` 的加密结果，或不传 Body。

```
GET /api/health/ping
Content-Type: text/plain
X-Session-Id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
X-Timestamp: 1719676800000
X-Nonce: QwE9rT2yU6iO0pA3sD5f==

Body: j3K9mP2xR7w...（空字符串 "" 加密后的 Base64）
```

#### 2.3.6 各方法 Body 明文格式速查

| HTTP 方法 | Body 明文格式 | 说明 |
|-----------|--------------|------|
| **GET** | `{"key1":"value1","key2":"value2"}` | Query 参数键值对 → JSON，无参数时为空字符串 |
| **POST** | `{"field1":"value1","nested":{"a":1}}` | 原始 JSON Body（对象/数组均可），无 Body 时为空字符串 |
| **POST（表单/文件）** | `{"field1":"value1","fileData":"BASE64..."}` | 表单字段 + 文件 Base64，合入一个 JSON |
| **PUT** | `{"field1":"newValue1",...}` | 原始 JSON Body，与 POST 规则一致 |
| **DELETE** | `{"id":"xxx","reason":"..."}` | 删除条件 → JSON，无参数时为空字符串 |

> **统一原则**：无论哪种 HTTP 方法，Body 的明文始终是 **合法的 JSON 字符串**（或空字符串），加密后按 `BASE64(IV \|\| CIPHERTEXT \|\| TAG)` 格式放入 Body，`Content-Type` 统一为 `text/plain`。服务端解密后根据 HTTP 方法和路径还原出原始语义——GET 还原为 Query String、POST/PUT 还原为 JSON Body、DELETE 还原为删除条件。

#### 2.3.7 密钥失效处理（协议约定）

业务请求发出后，密钥可能在以下几种情况下失效。协议层面约定了完整的**无感知恢复**流程。

##### 场景一：会话自然过期（最常见）

客户端发出请求时会话已过期（超过 30 分钟无使用 / 超过 1000 次请求），服务端解密时发现会话不存在。

**交互流程**：

```
客户端                                    服务端
   │                                        │
   ├── POST /api/payment/submit ────────────→│
   │   X-Session-Id: old-session-id          │
   │   Body: BASE64(IV || CIPHER || TAG)     │
   │                                        ├─ 查缓存：session 不存在/已过期
   │                                        ├─ 解密失败 → 不返回业务错误
   │   ←── HTTP 401 ────────────────────────┤
   │       X-Session-Expired: true           │
   │       Content-Type: application/json    │
   │       {"errorCode":"22301",             │
   │        "message":"会话不存在或已过期"}    │
   │                                        │
   ├─ ① 自动重新握手（3 次请求，见2.2）       │
   │   → 获得 new-session-id + 新 SM4 密钥   │
   │                                        │
   ├─ ② 用新会话重试原请求 ─────────────────→│
   │   POST /api/payment/submit             │
   │   X-Session-Id: new-session-id          │
   │   Body: BASE64(新加密的原 Body)          │
   │                                        ├─ 解密成功 → 正常处理
   │   ←── HTTP 200 ────────────────────────┤
   │       Body: BASE64(IV || CIPHER || TAG) │
   │                                        │
   └─ ③ 解密响应，返回业务结果                │
```

**HTTP 401 响应体**（Body 不加密，明文 JSON）：

```
HTTP 401 Unauthorized
Content-Type: application/json
X-Session-Expired: true

{
    "errorCode": "22301",
    "message": "会话不存在或已过期",
    "sessionId": null
}
```

> **注意**：401 响应的 Body 是**明文 JSON**，不经过 SM4 加密。因为此时会话密钥已失效，双方没有共同的加密密钥。客户端通过 `X-Session-Expired: true` + HTTP 401 组合判断应触发重握手。

---

##### 场景二：密钥在请求途中恰好失效（竞争窗口）

客户端发出请求时本地缓存显示会话还有 2 分钟，但请求到达服务端时刚好过期（或被其他并发请求耗尽次数）。处理方式与场景一完全相同——服务端返回 401，客户端自动重握手后重试。

**客户端实现要点**：

```
时间线：
  客户端本地会话剩余 120s  ──→ 发出请求
  请求到达服务端 ──────────→ 会话刚好过期（0s）
  服务端返回 401 ──────────→ 客户端捕获，触发重握手
  重握手完成 ──────────────→ 获取新会话
  重试请求 ────────────────→ 成功

总耗时：约 50ms（握手 3 次 HTTPS） + 原请求耗时
对调用方：完全透明，不抛异常
```

---

##### 场景三：TAG 校验失败（密钥不一致 / 数据被篡改）

服务端解密时 TAG 校验不通过，可能原因：
- 客户端和服务端的会话密钥不一致（客户端使用了已废弃的旧密钥）
- 数据在传输过程中被篡改
- 加密时使用的 AAD 与解密时不一致

**服务端响应**：

```
HTTP 400 Bad Request
Content-Type: application/json
X-Session-Expired: true

{
    "errorCode": "21202",
    "message": "SM4解密失败 - TAG校验失败",
    "sessionId": "a1b2c3d4-..."
}
```

**客户端处理**：TAG 校验失败意味着双方密钥状态不一致，**必须废弃当前会话，重新握手**。不能简单重试——重试用同一个密钥仍然会失败。

```
客户端收到 400 + errorCode=21202
  → ① 清除本地会话缓存（不尝试续期）
  → ② 重新 SM2 握手
  → ③ 用新会话重试原请求
```

---

##### 客户端自动重试的协议约定汇总

| 服务端返回 | 含义 | 客户端行为 | 是否重试原请求 |
|-----------|------|-----------|:---:|
| `HTTP 401` + `X-Session-Expired: true` | 会话过期/不存在 | 重握手 → 重试 | ✅ |
| `HTTP 400` + `errorCode=21202`（TAG 失败）| 密钥不一致 | 废弃会话 → 重握手 → 重试 | ✅ |
| `HTTP 400` + `errorCode=21103`（签名失败）| 握手阶段证书问题 | 检查本方密钥配置，**不重试** | ❌ |
| `HTTP 403` + Nonce 重复 | 重放攻击 | 生成新 Nonce 重试一次，再失败则告警 | ✅（仅一次） |
| `HTTP 200` + `X-Session-Renewed: {newId}` | 会话已被续期 | 更新本地 sessionId，后续请求用新 ID | N/A |

---

##### 重试的幂等性约束

对于 **GET / DELETE / PUT**（天然幂等），重试是安全的。

对于 **POST**（非幂等），客户端应在请求中携带 **幂等键**，防止重试导致重复创建资源：

```
客户端生成唯一请求 ID
  │
  ├── 首次请求 ──→ X-Idempotency-Key: req-20240101-001
  │                （Header 随 Body 一起被加密，对服务端透明）
  │
  └── 重试请求 ──→ X-Idempotency-Key: req-20240101-001（相同）
                   （服务端识别重复，返回首次处理结果）
```

| 方案 | 说明 |
|------|------|
| **客户端生成** | 每次 POST 请求生成一个 UUID 作为幂等键，加密在 Body JSON 的 `_idempotencyKey` 字段中 |
| **服务端识别** | 服务端解密后提取 `_idempotencyKey`，在 Redis 中缓存 `key → 响应结果`（TTL 5 分钟），相同 key 直接返回缓存结果 |
| **缓存命中** | 返回原始处理结果 + `X-Idempotent-Replay: true` Header，客户端可据此判断是幂等重放 |

> 协议层面**建议**所有 POST 请求的 JSON Body 中都预留 `_idempotencyKey` 字段。如果客户端不传，服务端按无幂等保护处理（重试可能导致重复创建）。

### 2.4 业务响应格式

```
HTTP 200
Content-Type: text/plain
X-Session-Expired: false

Body: BASE64( IV[12字节] || CIPHERTEXT || TAG[16字节] )
```

| Header | 含义 |
|--------|------|
| `X-Session-Expired: true` | 会话已过期，客户端应重新握手后重试 |
| `X-Session-Renewed: {newSessionId}` | 会话已续期，之后用新 sessionId |

### 2.5 错误响应格式

```json
{
    "errorCode": "21103",
    "message": "服务端证书验签失败",
    "sessionId": null
}
```

### 2.6 加密参数

| 参数 | 值 | 说明 |
|------|-----|------|
| SM4 密钥 | 16 字节 | KDF 派生，详见第三章 3.2 节 KDF 密钥派生规范 |
| SM4 模式 | GCM | |
| GCM IV | 12 字节 | 每次加密随机生成 |
| GCM TAG | 16 字节 | 自动校验 |
| AAD | `sessionId` 的 UTF-8 字节 | 加密和解密时必须一致 |

### 2.7 防重放

- 握手层：服务端校验 `timestamp` 偏差 ≤ 300 秒
- 业务层：服务端记录 `X-Nonce`，同一 Nonce 在 5 分钟内重复出现则拒绝（HTTP 403）

### 2.8 协议版本

握手 `protocolVersion` 字段用于协商。服务端不支持的版本返回 `HTTP 400`，客户端降级重试。

- [ ] SM4-GCM 加密：随机 IV → 加密 → Base64(IV + ciphertext + TAG)
- [ ] 每个请求 Header 注入 X-Session-Id + X-Timestamp + X-Nonce
- [ ] 收到 X-Session-Expired: true 时重新握手
- [ ] 握手时验证对方签名和时间戳
- [ ] Nonce 不重复，时间戳偏差不超过配置阈值

## 三、核心模块建设思路（SDK 实现参考）

### 3.1 会话管理模块

**设计目标**：安全存储会话密钥，支持多会话并发，自动清理过期会话。

**核心数据结构**：

- **本地缓存**：ConcurrentHashMap + 过期时间戳（用于SDK端）
- **分布式缓存**：Redis（用于服务端，支持多实例部署）
- **会话对象**包含：sessionId、SM4密钥、IV、创建时间、最后使用时间、剩余请求次数

**关键机制**：

- **惰性删除**：获取会话时检查是否过期，过期则自动删除并抛出异常
- **定时清理**：后台线程每60秒扫描一次，清理超时会话
- **续期策略**：剩余有效期小于5分钟时，自动发起续期请求（对业务方透明）
- **续期时密钥更新（Rekey）**：续期时不只延长时间，同时触发一次**轻量密钥更新**——复用已有的共享密钥 K，通过 KDF 派生新的 SM4 会话密钥和 IV（偏移量递增），避免同一密钥加密过多数据
- **总生命周期上限**：无论续期多少次，单个会话总存活时间不超过 **24小时**（可配置），达到上限后强制重新握手，防止长期使用同一组根密钥
- **限次策略**：单会话最多1000次请求，达到上限后强制自动重新握手

**并发控制**：

- 同一 sessionId 的并发请求加锁，防止重复握手
- 使用 `ConcurrentHashMap.computeIfAbsent` 实现一次性初始化

### 3.2 SM2密钥协商模块

**设计目标**：实现GB/T 32918.3-2016标准密钥交换协议，产出共享密钥。

**协议版本协商**：

握手第一条消息中携带协议版本号（如 `"1.0"`），服务端校验后决定是否兼容，防止双方独立升级导致不兼容。

**协议流程（3次握手，遵循标准第6章）**：

1. **客户端 → 服务端（第一消息）**：
   - 生成临时密钥对 `(rA, RA)`，其中 `RA = [rA]G`
   - 按标准 5.1.4 计算可辨别标识杂凑值：
     `ZA = SM3(ENTLA || IDA || a || b || xG || yG || xA || yA)`
     其中 `ENTLA` 为 IDA 的 2 字节长度，`(a, b)` 为椭圆曲线参数，`(xG, yG)` 为基点，`(xA, yA)` 为客户静态公钥
   - 使用客户静态私钥 `dA` 对 `(RA || IDA || ZA || timestamp)` 签名
   - 发送：`(RA, IDA, ZA, timestamp, signature, 协议版本号)`
2. **服务端 → 客户端（第二消息）**：
   - 验证时间戳偏差 ≤ 配置容差（默认300秒）
   - 使用客户静态公钥 `PA` 验证签名
   - 验证 `RA` 在 SM2 曲线上
   - 生成临时密钥对 `(rB, RB)`，其中 `RB = [rB]G`
   - 计算 `ZB`（同 ZA 计算方式，使用服务端标识）
   - 按标准计算共享密钥点：`(x1, y1) = [rB] * (RA + [x̄A] * PA)`，其中 `x̄A = 2^w + (xRA & (2^w - 1))`
   - 通过 KDF 派生会话密钥：`K = KDF(x1 || ZA || ZB, klen)`（见下文 KDF 规范）
   - 生成确认参数：`SB = SM3(0x02 || y1 || SM3(x1 || ZA || ZB || RA || RB))`
   - 发送：`(RB, ZB, SB)`
3. **客户端 → 服务端（第三消息）**：
   - 验证 `RB` 在 SM2 曲线上
   - 按标准计算共享密钥点：`(x1, y1) = [rA] * (RB + [x̄B] * PB)`，其中 `x̄B = 2^w + (xRB & (2^w - 1))`
   - 通过 KDF 派生会话密钥：`K = KDF(x1 || ZA || ZB, klen)`
   - 验证 `SB`：核对 `SB == SM3(0x02 || y1 || SM3(x1 || ZA || ZB || RA || RB))`
   - 生成确认参数：`SA = SM3(0x03 || y1 || SM3(x1 || ZA || ZB || RA || RB))`
   - 发送：`(SA)`，服务端验证后握手完成

**KDF 密钥派生规范（遵循 GB/T 32918.4-2016 第 5 章）**：

```
输入：Z = x1 || ZA || ZB （共享秘密的x坐标拼接双方杂凑值）
     klen = 要派生的密钥比特长度
输出：派生密钥
算法：
  1. 初始化计数器 ct = 0x00000001
  2. 对 i = 1 到 ⌈klen/v⌉：
       Hai = SM3(Z || ct)
       ct++
  3. 若 klen/v 有余数，对最后一个 Hai 取最左 klen-(v*⌊klen/v⌋) 位
  4. 输出：Ha1 || Ha2 || ... 的前 klen 位
其中 v = 256（SM3 输出长度）
```

**派生输出映射**：

| 用途            | 长度          | 偏移     |
|----------------|---------------|----------|
| SM4 会话密钥    | 128位（16字节）| 0        |
| SM4 IV（初始）  | 96位（12字节） | 128      |
| HMAC-SM3密钥   | 256位（32字节）| 224（可选）|

**关键参数**：

- 可辨别标识 `IDA`/`IDB`：双方各自的身份标识字符串（如系统名称+环境标识），用于计算 ZA/ZB
- `w`（椭圆曲线参数）：SM2 推荐曲线中 `w = ⌈log2(n)⌉ - 1`，实际取 `w = 127`
- 时间戳容差：默认300秒（可配置）
- 派生总长度 `klen`：默认 480 位（16 + 12 + 32 字节），含可选 HMAC 密钥

**异常处理**：

- 临时公钥不在曲线上 → 返回错误码，要求重新握手
- 共享密钥 x1 为 0（对应标准中无穷远点）→ 重新生成临时密钥对重试（最多3次）
- 密钥确认（SB/SA）验证失败 → 返回错误，客户端重新发起握手
- 时间戳偏差超限 → 返回错误码和服务器时间，供客户端校时

### 3.3 SM4加解密模块

**设计目标**：基于会话密钥，对业务数据进行认证加密。

**加密模式选择**：

- **强制使用GCM模式**（认证加密，自带完整性校验）
- 绝对禁止ECB模式
- 如环境不支持GCM，退化为CBC + HMAC-SM3组合模式

**加密参数固化**：

- SM4密钥长度：128位（16字节）
- GCM IV长度：96位（12字节）
- 认证标签长度：128位（16字节）
- 附加认证数据（AAD）：固定为 `sessionId.getBytes(StandardCharsets.UTF_8)`，AAD 不在密文中传输，由解密方从 `X-Session-Id` Header 提取拼接

**接口设计**（两层）：

| 接口层级           | 输入                 | 输出           | 说明                                          |
| :----------------- | :------------------- | :------------- | :-------------------------------------------- |
| **底层**           | key, iv, aad, plaintext | ciphertext+tag | 纯加密函数，无状态                            |
| **上层（业务用）** | sessionId, plaintext | Base64密文     | 自动从缓存取key+iv，AAD自动从sessionId构造，自动绑定sessionId到Header |

**传输格式约定**：

- 统一采用 `Base64(iv + ciphertext + tag)` 格式传输（AAD 通过 Header 的 `X-Session-Id` 隐式传递，无需重复编码）
- 确保 iv 每次都随机生成（即使同一个会话的多次加密，iv 也必须不同）
- 解码时按 12 + N + 16 字节拆分：前 12 字节为 IV，末 16 字节为 TAG，中间部分为密文

**防篡改**：

- 解密时自动校验 TAG，校验失败抛出 `IntegrityException`
- 建议在业务层捕获该异常，触发强制重握手

### 3.4 网络通信层

**设计目标**：对业务方透明，**仅指定的接口走加解密通道**，不影响项目内其他 HTTP 通信。

**核心思路**：

```
服务端                                   客户端
┌──────────────────────┐                ┌──────────────────────────┐
│ @Sm2Api              │  ← 加解密通道 → │ sm2Client.get("/api/..") │
│ public Result pay(..)│    SM4-GCM      │ sm2Client.post("/api/..")│
├──────────────────────┤                ├──────────────────────────┤
│ （无注解）            │  ← 明文直通 →  │ restTemplate.getForObject │
│ public Result health()│    不处理       │ （普通的 HTTP 调用）      │
└──────────────────────┘                └──────────────────────────┘
```

- **服务端**：通过 `@Sm2Api` 注解标记
- **客户端**：通过 `Sm2HttpClient` 工具类调用（`get`/`post`/`put`/`delete`），只有走这个工具类发出的请求才经过 SM2 处理
- 项目中其他 RestTemplate / Feign / OkHttp 调用完全不受影响

#### 3.4.1 `@Sm2Api` — 服务端暴露标记

```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface Sm2Api {
    /** 描述（可选，用于文档） */
    String value() default "";
}
```

**使用方式**：

```java
@RestController
public class PaymentController {

    // 方式1：标记在方法上 — 仅该方法需要加解密
    @Sm2Api
    @PostMapping("/api/payment/submit")
    public Result submit(@RequestBody PaymentRequest req) { ... }

    // 方式2：标记在类上 — 该 Controller 所有方法都需要加解密
    // （类上标记后，方法再标记 @Sm2Api(enabled=false) 可排除单个方法）
    @Sm2Api
    @RestController
    @RequestMapping("/api/sensitive")
    public class SensitiveController {

        @PostMapping("/query")
        public Result query(@RequestBody QueryRequest req) { ... }

        @PostMapping("/export")
        public Result export(@RequestBody ExportRequest req) { ... }
    }

    // 未标记 — 明文直通，过滤器不处理
    @GetMapping("/health")
    public String health() { return "OK"; }
}
```

**服务端过滤器执行流程**：

```
请求进入
  │
  ▼
过滤器：根据 URL 查找 HandlerMethod
  │
  ├─ 方法或类上有 @Sm2Api？
  │    │
  │    ├─ 是 → 提取 sessionId → 查缓存取密钥 → 解密 body → 替换为明文 → 放行到 Controller
  │    │                                    ↑ 响应时自动加密 ← Controller 返回后
  │    │
  │    └─ 否 → 直接放行（明文透传）
```

#### 3.4.2 客户端：`Sm2HttpClient` 工具类（推荐）

**设计原则**：不和项目现有的 HTTP 客户端（RestTemplate、OkHttp、Feign）抢地盘。`Sm2HttpClient` 是一个独立的工具类，只有主动调用它才会走 SM2 通道。

```java
// Sm2HttpClient 的核心 API
public class Sm2HttpClient {

    // GET 请求
    public Sm2Request get(String path) { ... }

    // POST 请求
    public Sm2Request post(String path) { ... }

    // PUT 请求
    public Sm2Request put(String path) { ... }

    // DELETE 请求
    public Sm2Request delete(String path) { ... }
}

// Sm2Request 链式构建器
public class Sm2Request {
    public Sm2Request param(String key, String value) { ... }
    public Sm2Request header(String key, String value) { ... }
    public Sm2Request body(Object jsonBody) { ... }
    public <T> T execute(Class<T> responseType) { ... }
}
```

**内部自动完成的流程**（每次 `execute()` 调用时）：

```
sm2Client.post("/api/payment/submit").body(data).execute(Result.class)
  │
  ▼
  1. 会话检查 → 无有效会话则自动 SM2 握手（含重试 + 熔断）
  2. 生成随机 IV → SM4-GCM 加密 body → Base64 编码
  3. 注入 Header：X-Session-Id + X-Timestamp + X-Nonce
  4. 通过 Hutool HttpUtil 发送 HTTPS 请求
  5. 校验响应 TAG → SM4-GCM 解密 body → 返回明文对象
  6. 会话快过期 → 自动后台续期（对调用方透明）
```

> 以上流程对调用方完全透明，业务代码只需调用 `sm2Client.get()/post()/put()/delete()` 即可，无需关心握手、加密、续期等细节。

#### 3.4.3 服务端过滤器

1. 根据请求 URL 查找 Spring HandlerMethod
2. 检查方法/类上是否有 `@Sm2Api`
3. 无注解 → 直接放行（明文透传）
4. 有注解 → **校验客户端对该接口的访问权限**（见 3.5 接口访问控制模块）
5. 从 Header 提取 `X-Session-Id`，解析客户端身份
6. 从缓存获取会话密钥
7. 解密 RequestBody，替换为明文供 Controller 使用
8. Controller 返回后，加密响应写回客户端

> **设计要点**：路径匹配（3.4.1 之前的 include/exclude）降级为**辅助配置**，用于对第三方接口（不受控、无法加注解）的兜底场景。正常情况下以注解为准。

### 3.5 接口访问控制模块

**设计目标**：不同客户端有不同接口访问权限，由服务端统一管控，支持配置文件与数据库两种数据源。

**核心思路**（与 `@Sm2Api` 配合）：

```
        │
        ▼

     │                │

     │                │

  直接放行     1. 提取 X-Session-Id

               3. 调用 accessProvider.check(clientId, path, method)

                       ▼
```


**SPI 接口定义**：

```java
// 服务端提供的扩展点接口
public interface ApiAccessProvider {

    /**
     * 检查客户端是否有权访问指定接口
     * @param clientId   客户端标识（握手时认证）
     * @param path       请求路径（如 /api/user/query）
     * @param method     HTTP 方法（GET/POST/PUT/DELETE）
     * @return true=允许访问, false=拒绝
     */
    boolean check(String clientId, String path, String method);

    /**
     * 获取某个客户端可访问的完整路径列表（用于管理后台展示）
     */
    List<String> getAllowedPaths(String clientId);

    /**
     * 刷新/重新加载配置（实现类自行决定是否支持）
     */
    default void refresh() {}
}
```

**默认实现（配置文件方式）**：

```java
public class ConfigBasedApiAccessProvider implements ApiAccessProvider {

    // 配置结构
    // sm2.access.client-A.paths = /api/user/**, /api/order/query
    // sm2.access.client-B.paths = /api/public/**

    private Map<String, List<PathMatcher>> clientPathMap;

    @PostConstruct
    public void loadConfig() {
        // 从配置中心/本地 yml 加载各客户端允许的路径
        // 支持 Ant 风格通配符：/api/user/**  /api/order/query*
        // 监听配置变更事件，自动刷新 clientPathMap
    }

    @Override
    public boolean check(String clientId, String path, String method) {
        List<PathMatcher> matchers = clientPathMap.get(clientId);
        if (matchers == null) return false;  // 未配置的客户端默认拒绝
        return matchers.stream().anyMatch(m -> m.matches(path));
    }
}
```

**配置文件示例（application.yml）**：


**自定义实现（数据库方式）**：

```java
// 业务方实现此接口，注入 Spring 容器即可自动替换默认实现
@Component
public class DatabaseApiAccessProvider implements ApiAccessProvider {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    private LoadingCache<String, Set<String>> pathCache = Caffeine.newBuilder()
        .expireAfterWrite(60, TimeUnit.SECONDS)  // 1分钟本地缓存
        .build();

    @Override
    public boolean check(String clientId, String path, String method) {
        Set<String> allowed = pathCache.get(clientId, this::loadFromDb);
        return allowed.stream().anyMatch(p -> pathMatch(p, path));
    }

    private Set<String> loadFromDb(String clientId) {
        return jdbcTemplate.queryForList(
            "SELECT path_pattern FROM api_access_rule WHERE client_id = ? AND enabled = 1",
            String.class, clientId
        ).stream().collect(Collectors.toSet());
    }

    // 数据库表结构参考：
    // CREATE TABLE api_access_rule (
    //     id BIGINT PRIMARY KEY AUTO_INCREMENT,
    //     client_id VARCHAR(64) NOT NULL COMMENT '客户端标识',
    //     path_pattern VARCHAR(256) NOT NULL COMMENT '路径匹配模式，支持Ant风格',
    //     http_method VARCHAR(10) DEFAULT '*' COMMENT '允许的HTTP方法',
    //     enabled TINYINT DEFAULT 1,
    //     created_at DATETIME,
    //     updated_at DATETIME,
    //     INDEX idx_client (client_id)
    // );
}
```

**接入流程**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 业务方实现 `ApiAccessProvider` | 默认用配置文件，需要动态管理则实现数据库版本 |
| 2 | 注册到 Spring 容器 | SDK 自动检测，存在自定义 Bean 则覆盖默认实现 |
| 3 | 服务端过滤器调用 `check()` | 每次请求自动校验，对客户端透明 |
| 4 | 权限变更 | 配置文件版支持热更新；数据库版实时生效（含本地缓存） |

**与握手的配合**：

- 握手阶段已认证客户端身份（通过 SM2 签名验证 `PA`）
- `clientId` 在握手成功后写入 Session 对象
- 服务端过滤器从 Session 中取出 `clientId`，无需客户端额外传递

## 四、工程落地关键点

### 4.1 密钥管理

**长期密钥（静态，不变）**：

每个参与方生成一对 SM2 公私钥，部署后**长期保持不变**。对接时双方交换公钥。

```
系统 A：生成 (dA, PA)  ──→ 把 PA 给 B
系统 B：生成 (dB, PB)  ──→ 把 PB 给 A
```

| 密钥 | 存储位置 | 说明 |
|------|----------|------|
| 本方私钥 | 配置文件 / 环境变量 / 密钥文件 | 生产环境禁止硬编码在代码中 |
| 对方公钥 | 配置文件（`peer.xxx.public-key`）| 部署时配好，不改 |

> 如果需要更换密钥（如密钥泄露），更新配置文件后重启即可。会话密钥由每次握手的临时密钥对保证安全性，与长期密钥无关。

**临时密钥（每次会话自动生成）**：

- 临时私钥 `rA`、`rB`：仅存在于内存，握手完成后立即置零
- 共享密钥 `K`：仅存在于内存，**不落地、不入库、不打日志**
- 每次握手生成全新临时密钥对，保证**前向安全性**

**Java 内存安全处理**：

> ⚠️ `byte[]` 在 Java 中无法彻底安全擦除：JVM GC 可能复制对象留下密钥残留，JIT 可能优化掉"无用"的 `Arrays.fill()` 操作。

| 防护等级 | 措施 | 适用场景 |
|----------|------|----------|
| **基础**（推荐） | 使用 `char[]`，用后 `Arrays.fill(key, '\0')` | 一般业务系统 |
| **增强** | `ByteBuffer.allocateDirect()` 堆外内存 | 金融/政务系统 |
| **最高** | HSM 硬件密码机 / SGX 飞地 | 等保三级以上 |

**会话密钥（工作级）**：

- 本方发起的会话（主动调用）：缓存在本地 Caffeine 缓存
- 对方发起的会话（被动响应）：缓存在 Redis（多实例共享，键 `sm2:session:{sessionId}`，TTL 30分钟）
- Redis 中的 Session 对象用本地生成的随机密钥做 SM4 加密后存储，密钥仅存在于内存，重启后旧会话自动失效

### 4.2 防重放攻击设计

**双重校验机制**：

| 层级       | 机制       | 实现方式                                                     |
| :--------- | :--------- | :----------------------------------------------------------- |
| **握手层** | 时间戳校验 | 握手消息携带时间戳，服务端校验偏差≤300秒                     |
| **业务层** | Nonce校验  | 每个请求携带16字节随机Nonce，服务端存入Redis（有效期5分钟），同一Nonce重复使用则拒绝 |

**Nonce管理**：

- 客户端生成：每次请求生成新的16字节随机数
- 基础方案：`Redis SETNX sm2:nonce:{nonce} 1 EX 300`，SETNX返回0时拒绝请求
- 高并发优化方案（推荐）：

```
高并发 Nonce 校验方案（Bloom Filter + 精确校验两级架构）：

第一级 — 内存 Bloom Filter：
  - 在服务端内存维护 Bloom Filter，哈希函数使用 SM3
  - 每分钟一个 Filter（key: sm2:nonce:bf:{minute_tick}），保留最近 5 分钟
  - 新 Nonce 先过 Bloom Filter：不存在则直接放行（极小概率误判）
  - 可能存在则进入第二级精确校验

第二级 — Redis 精确校验（兜底）：
  - Bloom Filter 报告"可能存在"时，再查 Redis 精确判断
  - 由于 Bloom Filter 已拦截绝大部分请求，Redis 压力大幅降低

内存估算（假设 10,000 QPS）：
  - 5 分钟 × 60 万 Nonce，Bloom Filter 0.1% 误判率
  - 每个 Filter 约需 ~9MB，5 个共 ~45MB（JVM 堆可承受）
  - Redis 中仅需存储 Bloom 误判穿透的 ~3000 条，而非 300 万条
```

### 4.3 超时与重试机制

**握手超时**：

- 单次握手超时时间：5秒（网络抖动时可配置）
- 重试策略：失败后最多重试3次，间隔递增（1s, 2s, 4s）

**会话过期处理**：

- 业务请求时发现会话过期 → 自动触发重握手 → 重试原请求（对业务方透明）
- 重握手失败 → 抛出`SessionEstablishException`，业务方自行决定降级或报错

**熔断保护**：

- 连续握手失败5次 → 熔断器打开，直接返回错误（避免雪崩）
- 熔断时间窗口：30秒后尝试恢复

**降级与逃生策略**：

| 场景                       | 降级策略                                                     | 恢复条件                     |
| -------------------------- | ------------------------------------------------------------ | ---------------------------- |
| **KMS 不可用**             | 使用本地加密存储的 DEK 缓存文件（用服务端 SM2 公钥加密），继续为已有和新建会话提供服务 | KMS 恢复后自动切回           |
| **Redis 不可用**           | 服务端降级为本地内存缓存（单机模式），配合负载均衡会话保持使用 | Redis 恢复后切回分布式缓存   |
| **握手服务整体异常**       | 返回 `HandshakeFailedException`，业务方可选择：① 返回降级响应；② 使用上次成功会话的密钥（需配置 `fallback.enabled=true`，有效期与上次会话一致） | 握手成功率恢复后自动关闭降级 |
| **Nonce 校验 Redis 故障**  | 仅依赖时间戳校验（降低安全级别），降级期间记录告警           | Redis 恢复后恢复双重校验     |
| **极端逃生（人工审批）**   | 管理员通过配置开关临时关闭加密，明文传输（所有操作留审计日志，自动触发告警，最大开启时间限制 10 分钟） | 故障修复后立即关闭           |

所有降级和逃生事件自动触发 P1 级别告警，并在审计日志中记录完整操作链。

## 六、日志与监控设计

### 6.1 日志规范

**必须记录的日志**（脱敏后）：

- 握手开始/成功/失败（含sessionId，不含密钥）
- 会话创建/续期/销毁
- 加解密异常（含sessionId、错误码）
- 防重放拦截记录（含nonce）

**严禁记录的日志**：

- SM2私钥（静态或临时）
- 共享密钥K
- SM4会话密钥
- 业务明文数据

**日志脱敏示例**：

```text
✅ 正确：sessionId=abc123, 协商成功, 耗时=12ms
❌ 错误：sessionId=abc123, key=3F7A8B..., iv=9E2D...
```



### 6.2 监控指标

建议暴露以下Metrics（用于Prometheus等监控系统）：

- `sm2_handshake_total`：握手总次数（按结果状态区分成功/失败）
- `sm2_handshake_duration`：握手耗时分布
- `sm2_session_active`：当前活跃会话数
- `sm2_encrypt_requests`：加密请求总数
- `sm2_decrypt_errors`：解密失败次数（区分TAG校验失败/会话过期）
- `sm2_replay_blocks`：防重放拦截次数

### 6.3 告警规则

| 告警项           | 阈值             | 处理建议           |
| :--------------- | :--------------- | :----------------- |
| 握手失败率 > 10% | 最近5分钟        | 检查网络/密钥配置  |
| TAG校验失败突增  | 最近1分钟 > 10次 | 可能遭遇篡改攻击   |
| 会话数超过上限   | > 10000          | 检查是否有会话泄漏 |
| Nonce重复率 > 1% | 最近5分钟        | 可能遭遇重放攻击   |

## 七、安全加固清单

### 7.1 编译期安全

- **代码混淆**：SDK发布时进行混淆，降低逆向分析风险
- **字符串加密**：敏感字符串（如算法标识）在编译期加密
- **完整性校验**：SDK Jar包发布时附带MD5/SHA256校验值

### 7.2 运行期安全

- **内存清理**：临时密钥使用后立即 `Arrays.fill(key, (byte)0)`（参见 4.1 节内存安全分层防护）
- **堆转储控制**：生产环境建议配置 `-XX:+DisableAttachMechanism` 防止通过 jmap 等工具 dump 内存；但需同时提供**运维应急开关**（如通过配置中心动态关闭），允许排查线上故障时临时启用 Attach 机制，所有 Attach 操作记录审计日志
- **加密验签**：SDK 自身 Jar 包启动时验签，防止被篡改（验签公钥硬编码在 SDK 内部，通过代码混淆保护）

### 7.3 通信安全

- **TLS叠加**：即使已有SM4加密，建议仍走HTTPS（双重加密）
- **HSTS**：服务端开启HSTS，强制HTTPS
- **证书固定**（Pinning）：SDK内置服务端证书指纹，防止中间人劫持握手过程
  - **多指纹备份**：SDK 至少内置 **2 个备用公钥指纹**（当前证书 + 预置的下一个证书），证书到期换证时免去客户端同步升级
  - **动态更新**：支持通过配置中心下发新的证书指纹列表，客户端启动时拉取合并

## 二、使用指南

> **一个 SDK，两个角色**：下面的示例以"系统 A 对接系统 B"为例。实际使用时，双方各自引入 SDK，根据自身角色配置相应部分即可。

### 2.1 主动调用（作为客户端）：`Sm2HttpClient`

**设计思路**：`Sm2HttpClient` 内部使用 **Hutool `HttpUtil`** 发送请求，自动完成握手、加密、解密。只有走 `sm2Client.get()`/`post()` 的请求才经过 SM2，项目里其他 HTTP 调用完全不受影响。

**只需配 yml，然后调用**：

```yaml
# application.yml — 系统 A 侧配置
sm2-sdk:
  # 本方身份
  my-identity: SYSTEM_A
  my-private-key: /secure/system_a_private.pem
  # 对方（系统 B）的公钥和服务地址
  peer:
    SYSTEM_B:
      public-key: /secure/system_b_public.pem
      server-url: https://api.system-b.com
  # 主动调用角色：开启
  client:
    enabled: true
  # 被动响应角色：如果系统 B 也会调用本系统，则也开启
  server:
    enabled: true
    redis:
      host: ${REDIS_HOST:127.0.0.1}
      port: ${REDIS_PORT:6379}
```

```java
// 注入 Sm2HttpClient（SDK 自动配置好的 Bean）
@Autowired
private Sm2HttpClient sm2Client;

// ====== GET 请求 ======
UserInfo user = sm2Client
    .get("/api/user/query")
    .param("idCard", "110101199001011234")
    .execute(UserInfo.class);

// ====== POST 请求（JSON body） ======
SubmitResult result = sm2Client
    .post("/api/payment/submit")
    .body(new PaymentRequest(order))
    .execute(SubmitResult.class);

// ====== POST 请求（带自定义 Header） ======
Result r = sm2Client
    .post("/api/order/create")
    .header("X-Request-Id", requestId)
    .body(orderRequest)
    .execute(Result.class);

// ====== PUT 请求 ======
sm2Client.put("/api/user/update")
    .body(updateRequest)
    .execute(Void.class);

// ====== DELETE 请求 ======
sm2Client.delete("/api/cache/clear")
    .param("scope", "all")
    .execute(Void.class);

// ====== 普通 HTTP 请求（不走 SM2，和原来一样） ======
// 用 RestTemplate / OkHttp / Feign 正常发请求即可
String health = restTemplate.getForObject("https://api.partner.com/health", String.class);
```

**内部自动完成的事情**（业务代码无感知）：

| 步骤 | 行为 |
|------|------|
| ① 会话检查 | 无有效会话 → 自动发起 SM2 握手 |
| ② 加密 | 请求体自动 SM4-GCM 加密，IV 每次随机 |
| ③ 注入 Header | 自动添加 `X-Session-Id`、时间戳、Nonce |
| ④ 发送 | 通过 Hutool `HttpUtil` 发送 HTTPS 请求 |
| ⑤ 解密 | 响应体自动 SM4-GCM 解密 + TAG 校验 |
| ⑥ 续期 | 会话快过期时自动续期（对业务透明） |

### 2.2 被动响应（作为服务端）：`@Sm2Api`

**在自己的 Controller 上加 `@Sm2Api`，声明"这个接口可以被对方系统通过 SM2 加密调用"**。SDK 过滤器自动解密请求、加密响应。未加注解的方法不受影响。

```java
@RestController
public class PaymentController {

    @Sm2Api   // ← 对方系统通过 sm2Client 调此接口时，SDK 自动解密
    @PostMapping("/api/payment/submit")
    public Result submit(@RequestBody PaymentRequest req) {
        // req 已经是解密后的明文
        return paymentService.process(req);
    }

    // 不加注解 → 内部接口，SM2 过滤器直接放行
    @GetMapping("/internal/health")
    public String health() { return "OK"; }
}
```

**配置**（和主动调用使用同一个 yml，只需开启 `server.enabled`）：

```yaml
sm2-sdk:
  my-identity: SYSTEM_A
  my-private-key: /secure/system_a_private.pem
  peer:
    SYSTEM_B:
      public-key: /secure/system_b_public.pem
  server:
    enabled: true
    redis:                              # 分布式会话（多实例部署时需要）
      host: 127.0.0.1
```

### 2.3 其他用法

**注解方式（Feign）**— 适合用 Feign 声明式调用的项目：

```java
@FeignClient(name = "payment", url = "${sm2-sdk.server-url}")
public interface PaymentClient {

    @Sm2Call
    @PostMapping("/api/payment/submit")
    SubmitResult submit(@RequestBody PaymentRequest req);

    // 不加注解 → 普通请求
    @GetMapping("/api/payment/status")
    StatusResult status(@RequestParam String orderId);
}
```

**纯手动模式**— 适合非 Spring 项目、Android、CLI 工具：

```java
Sm2HttpClient client = Sm2HttpClient.create(SdkConfig.builder()
    .clientIdentity("CLIENT_001")
    .serverIdentity("SERVER_PAYMENT")
    .clientPrivateKeyPem("path/to/client_private.pem")
    .serverPublicKeyPem("path/to/server_public.pem")
    .serverUrl("https://api.yourcompany.com")
    .build());

// 调用方式和注入 Bean 完全一样
UserInfo user = client.get("/api/user/query").param("id", "123").execute(UserInfo.class);
```

> **推荐工具类方式**：不依赖 AOP、不修改拦截器、不影响其他 HTTP 调用。只有 `sm2Client.get()/post()/put()/delete()` 发出的请求走 SM2 通道，项目中其他 RestTemplate/Feign/OkHttp 调用原封不动。

### 2.4 异常处理示例

```java
try {
    sdkClient.initSession();
    String encrypted = sdkClient.encrypt(sessionId, plainText);
} catch (SessionExpiredException e) {
    // 会话过期，自动重握手（SDK内部已自动处理，但业务方可自定义）
    log.warn("会话已过期，重新建立: {}", e.getSessionId());
    sdkClient.refreshSession();
} catch (HandshakeFailedException e) {
    // 握手失败，可能需要告警
    log.error("密钥协商失败，错误码: {}, 原因: {}", e.getErrorCode(), e.getMessage());
    // 降级策略：使用备用通道或返回友好提示
    return fallbackResponse();
} catch (IntegrityException e) {
    // 数据完整性校验失败（可能被篡改）
    log.error("数据完整性校验失败: {}", e.getMessage());
    // 要求重新请求
    throw new SecurityException("数据传输异常，请重试");
} catch (Sm2SdkException e) {
    // 通用异常，记录错误码和追踪ID
    log.error("SDK异常, errorCode={}, traceId={}", e.getErrorCode(), e.getTraceId());
    throw e;
}
```

```java
// 在 Controller 或方法上加 @Sm2Api，声明"此接口需要 SM2 加解密 + 访问控制"
// 不加注解的方法 → SDK 过滤器自动跳过，明文传输

@RestController
public class PaymentController {
```

#### 2.6.4 注册过滤器（自动，无需手动配置）

```java
// SDK 通过 spring.factories 自动加载 Sm2ServerAutoConfiguration
// 自动注册 Sm2ServerFilter，自行检测 HandlerMethod 上的 @Sm2Api 注解
// 正常情况下无需编写以下代码

// 仅在需要自定义过滤器顺序时手动声明：
@Configuration
public class ServerConfig {
    @Bean
    public FilterRegistrationBean<Sm2ServerFilter> sm2Filter(Sm2ServerSdk sdk) {
        FilterRegistrationBean<Sm2ServerFilter> bean = new FilterRegistrationBean<>();
        bean.setFilter(new Sm2ServerFilter(sdk));
        bean.addUrlPatterns("/api/*");     // 仅对 /api 下的路径生效
        bean.setOrder(Ordered.HIGHEST_PRECEDENCE + 10);
        return bean;
    }
}
```

```java
// 方式一：继续用 yml 配置 — 无需任何代码，已在 2.5.2 配置完成

// 方式二：从数据库加载 — 实现接口并注册为 Spring Bean，SDK 自动替换
@Component
public class MyDbAccessProvider implements ApiAccessProvider {
    @Autowired
    private ApiAccessRuleMapper mapper;  // 你的 Mapper
```

| 步骤 | 服务端 | 客户端 |
|------|--------|--------|
| ① 加依赖 | `sm2-server-sdk` | `sm2-client-sdk` |
| ② 配 yml | 私钥 + 客户端公钥列表 + Redis + 访问控制规则 | 客户端标识 + 自身私钥 + 服务端公钥 + 服务端 URL |

### 2.7 Spring Boot 自动配置设计

SDK 只提供一个 `Sm2SdkAutoConfiguration`，根据 yml 配置自动决定开启哪些能力：

```java
@Configuration
@ConditionalOnProperty(prefix = "sm2-sdk", name = "my-identity")
public class Sm2SdkAutoConfiguration {

    private final Sm2SdkProperties props;

    // ============ 核心 Bean：始终创建 ============

    @Bean
    @ConditionalOnMissingBean
    public Sm2Sdk sm2Sdk(Sm2SdkProperties props) {
        return Sm2Sdk.create(props);  // 统一入口，管理密钥、会话、配置
    }

    // ============ 主动调用：client.enabled=true 时创建 ============

    @Bean
    @ConditionalOnProperty(prefix = "sm2-sdk.client", name = "enabled", havingValue = "true")
    public Sm2HttpClient sm2HttpClient(Sm2Sdk sdk) {
        // 内部使用 Hutool HttpUtil 发送请求
        // 业务方 @Autowired 注入，调 get()/post()/put()/delete()
        return sdk.createHttpClient();
    }

    // ============ 被动响应：server.enabled=true 时创建 ============

    @Bean
    @ConditionalOnProperty(prefix = "sm2-sdk.server", name = "enabled", havingValue = "true")
    public FilterRegistrationBean<Sm2ServerFilter> sm2ServerFilter(Sm2Sdk sdk) {
        return new FilterRegistrationBean<>(new Sm2ServerFilter(sdk));
    }

    @Bean
    @ConditionalOnProperty(prefix = "sm2-sdk.server", name = "enabled", havingValue = "true")
    @ConditionalOnMissingBean(ApiAccessProvider.class)
    public ApiAccessProvider defaultApiAccessProvider(Sm2SdkProperties props) {
        return new ConfigBasedApiAccessProvider(props.getAccess());
    }
}
```

**配置属性映射**：

```java
@ConfigurationProperties(prefix = "sm2-sdk")
public class Sm2SdkProperties {
    private String myIdentity;              // 本方标识
    private String myPrivateKey;            // 本方 SM2 私钥路径
    private Map<String, PeerConfig> peers;  // 对方系统配置（peerId → 配置）
    private ClientConfig client;            // 主动调用配置
    private ServerConfig server;            // 被动响应配置
    private AccessConfig access;            // 访问控制

    public static class PeerConfig {
        private String publicKey;           // 对方 SM2 公钥路径
        private String serverUrl;           // 对方服务地址
    }
    public static class ClientConfig {
        private boolean enabled = false;
    }
    public static class ServerConfig {
        private boolean enabled = false;
        private Redis redis;
    }
    public static class AccessConfig {
        private String defaultPolicy = "deny";
        private Map<String, ClientAccess> peers;
    }
    public static class ClientAccess {
        private List<String> paths;
    }
}
```

**双方都接入后，一次调用全链路**：

```
系统 A                               系统 B
──────                               ──────
sm2Client.post("/api/pay")           @Sm2Api → /api/pay
  │                                    │
  ├─ 自动握手（SM2 密钥协商）────────────┤
  ├─ 加密 body（SM4-GCM）               │
  ├─ Hutool HttpUtil 发送 ────────────→ │
  │                                    ├─ Filter 检测 @Sm2Api
  │                                    ├─ 查 access 权限
  │                                    ├─ 解密 body → Controller
  │                                    ├─ 加密 response
  │  ←───────────────────────────────── │
  ├─ 解密 response                      │
  └─ 返回 Result 对象                   │
```

**接入：一个依赖 + 一份配置**：

```
① pom.xml 加一个依赖：sm2-sdk
② application.yml 配本方身份 + 密钥 + 对方公钥 + 按需开启 client/server
③ 主动调用：注入 Sm2HttpClient 调 get()/post()
   被动响应：在 Controller 上加 @Sm2Api
④ 启动 ✅
```

> **设计原则**：SDK 负责所有加解密、握手、会话、鉴权的复杂逻辑；业务方只需配 yml，需要主动调用就注入 `Sm2HttpClient`，需要被调用就加 `@Sm2Api`。

## 三、异常码规范

### 3.1 异常码分类规范

采用**5位数字编码**，格式：`ABC`（模块）+ `DD`（具体错误）

```
格式: A B C D D
      │ │ │ └── 具体错误序号（00-99）
      │ │ └──── 操作类型（1=握手/2=加解密/3=会话/4=配置/5=网络/9=通用）
      │ └────── 模块层级（1=客户端/2=服务端）
      └──────── 错误级别（1=警告/2=严重/3=致命）
```

### 3.3 异常码使用示例（客户端）

```java
// 异常类设计
public class Sm2SdkException extends RuntimeException {
    private final String errorCode;    // 5位错误码
    private final String traceId;      // 分布式追踪ID
    private final String sessionId;    // 关联的会话ID（如有）
    
    public Sm2SdkException(ErrorCode errorCode, String message) {
        super(message);
        this.errorCode = errorCode.getCode();
        this.traceId = TraceContext.getTraceId();
    }
}

// 错误码枚举
public enum ErrorCode {
    HANDSHAKE_SIGN_VERIFY_FAILED("21103", "服务端签名验证失败"),
    SESSION_EXPIRED("11301", "会话已过期，请重新握手"),
    // ...
}

// 业务代码捕获
try {
    sdkClient.encrypt(sessionId, data);
} catch (Sm2SdkException e) {
    switch (e.getErrorCode()) {
        case "11301":  // 会话过期
            sdkClient.refreshSession();
            // 重试加密
            break;
        case "21202":  // TAG校验失败
            // 安全事件，记录告警
            securityAlert(e);
            throw new DataIntegrityException("数据可能被篡改");
        default:
            throw e;
    }
}
```
